# ADR-016: 教师指令改存 SCF 内存变量，不再走飞书多维表格

## 状态
已采纳（2026-04-20）

## 背景
教师后台发布课程后，学生端通过 GitHub Pages 收不到指令。排查发现：
1. 飞书免费版 API 调用上限 1 万次/月，13台 iPhone 每30秒轮询 + 数据上报，两周就超额
2. 超额后飞书返回 `99991403`，但 SCF 代码吞掉了错误，导致 set "成功" 但 get 返回 null
3. 教师指令（teacherCommand、currentLesson）是临时数据，不需要持久化到数据库

## 决策
- **教师指令（teacherCommand、currentLesson）** 改存 SCF 云函数的全局变量 + `/tmp` 文件双保险
- **学生行为日志、成绩数据** 继续走飞书多维表格（重要数据需持久化）
- 这样轮询不再消耗飞书 API 配额，月调用量从 ~44,600 次降到 ~3,000 次，免费额度够用

## 否定的方案

### 方案1：升级飞书商业版
- 50元/人/月起，对一个博士研究项目来说太贵
- 杀鸡用牛刀，教师指令根本不需要数据库级持久化

### 方案2：换其他云存储（COS、Redis）
- 增加额外服务依赖和配置复杂度
- SCF 内存变量对低流量场景完全够用

### 方案3：只用本地服务器（局域网 /api 内存存储）
- 已实现为开发环境方案（firebase-sync.js 的 _isLocal 判断）
- 但教学场景需要 GitHub Pages 方式（学生 iPhone 收藏的网址），不能依赖教师每次开电脑当服务器

## 风险与缓解
- **风险**：SCF 多实例导致内存不共享 → **缓解**：`/tmp` 文件备份，写入时同时写内存和文件，读取时先内存后文件
- **风险**：SCF 冷启动丢失数据 → **缓解**：教师重新点"发布"即可，指令数据本身是幂等的
- 实际风险很低：13台手机流量只需1个实例，30秒轮询保持实例活跃

## 成本影响
- SCF 免费额度：100万次/月、40万 GB-s/月
- 当前用量：~4.5万次/月、~572 GB-s/月
- **零额外费用**

## 技术细节
- 修改 `tencent_scf.js`：`set`/`get` action 对 teacherCommand 和 currentLesson 使用全局变量 + /tmp 文件
- 修改 `firebase-sync.js`：本地网络（192.168.x.x）走 `/api` 内存存储（已完成 2026-04-20）
- SCF 的 `set` 返回值应传回飞书实际响应，不再吞错误
