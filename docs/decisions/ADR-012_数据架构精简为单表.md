# ADR-012: 数据架构精简为单表 app_behavior_log

## 状态
已采纳（2026-04-13）

## 背景
三张飞书表（app_behavior_log、student_profile、pair_collab_log）导致大量字段类型适配 bug，花费 3+ 小时调试。每多一张表就多一个故障点——SOE 字段 null→Number 类型不匹配、upsert 去重逻辑复杂、三张表各自的写入时序问题，全部叠加。

## 决策
精简为只保留 **app_behavior_log 一张表**。student_profile 和 pair_collab_log 的写入逻辑全部改为 no-op（保留函数壳，调用方不报错）。

## 理论依据（First Principles）

1. **student_profile 100% 冗余**：所有字段（totalQuestions, correctCount, speakingCount 等）均可从 behavior_log 按 studentId 聚合得出。写两份数据=双倍维护成本，零额外信息。
2. **pair_collab_log 冗余**：除非论文专门分析合作质量（如 A/B 角色贡献比），否则 behavior_log 已包含 pairId + asRoleA，足够聚合。
3. **每多一张表=多一个故障点**：三张表花了 3 小时修类型问题（见 CHANGELOG v=105）。单表=单一数据源=单一故障点。
4. **四个研究目标只需 behavior_log + 飞书已有的纸笔测试表**：
   - Obj1（设计 App）：不依赖数据表
   - Obj2（英语发展）：纸笔测试 pre/post
   - Obj3（参与度）：behavior_log 的 timeSpentMs, retryCount, speakingCount
   - Obj4（学习成果）：behavior_log 的 isCorrect, scaffoldLevelUsed, soeScore

## 保留字段
logId, studentId, partnerId, pairId, classId, sessionDate, lessonId, stationId, questionType, questionIndex, isCorrect, asRoleA, timeSpentMs, scaffoldLevelUsed, retryCount, soeScore

## 移除字段
starRewarded, stationCompleted, stationStarRating, clientVersion, soePronAccuracy, soePronFluency, soePronCompletion

## 专家团审查
导师/教师/测试专家/产品/学生五角色一致同意。

## 对 Dashboard 的影响

分析了 `tencent_scf.js` 的 `queryProgress` handler 和 `teacher/dashboard.html`，发现：

1. **student_profile 表：有冲突但不致命。** queryProgress 第 351-365 行会读 student_profile 表来构建 studentId→name 映射（用于在 dashboard 上显示学生姓名）。但：
   - 代码有 `try/catch` 包裹，读取失败不会导致 dashboard 崩溃
   - 有兜底逻辑：优先从 behavior_log 记录的 `studentA`/`studentB` 字段取名字，再兜底为"组{pairId}"
   - **结论：停止写入 student_profile 后，dashboard 仍能正常工作，只是学生姓名可能显示为组号。如果 behavior_log 的记录中包含 studentA/studentB 字段则无影响。**

2. **pair_collab_log 表：无冲突。** queryProgress 完全不查询此表。

3. **starRewarded 字段：有冲突。** queryProgress 第 397 行用 `f.starRewarded` 计算星星数。移除此字段后，dashboard 上已完成小组的星星数将始终显示为 0。**需要后续修改 queryProgress 逻辑，改为从 isCorrect 派生星星数，或移除星星展示。**

## 风险
- 如果后续需要合作质量分析（如论文 Obj3 深入分析 A/B 角色贡献差异），需要重新开启 pair_collab_log
- Dashboard 星星数显示暂时为 0，需要跟进修复 queryProgress 的 starRewarded 逻辑

## 论文怎么写
To minimize technical complexity and data inconsistency, the app employs a single-table architecture (app_behavior_log) where each row represents one question-level interaction. Aggregate metrics (e.g., per-student accuracy, pair collaboration patterns) are computed from this log at analysis time rather than maintained in separate tables, following the principle of a single source of truth.
