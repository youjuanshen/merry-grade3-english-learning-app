## 一、词汇管理规范

### 规则

**题库所有词汇（包括干扰项）必须在本课词汇列表内，不能出现生词。**

具体要求：
- 每课开发前，先建立本课词汇表（参考 `知识点清单_U1L1.md` 格式）
- 题目的 `correct` 字段和所有 `distractor`（干扰项）字段，逐一对照词汇表核查
- 口语题的跟读目标词、阅读题的配对选项，同样适用此规则

### 反例

U1L1 开发时出现了以下词汇超出本课范围的情况（修复耗时约2小时）：

| 位置 | 问题词汇 | 原因 | 处理 |
|------|---------|------|------|
| 听力站2 第2题 | monkey | 上学期知识点，不属于 U1L1 | 替换为本课词汇 |
| 听力站2 第3题 | duck | 上学期知识点，不属于 U1L1 | 替换为本课词汇 |
| 听力站3 第4题 | duck + monkey | 同上 | 替换为本课词汇 |
| 阅读站1 第5题 | duck | 同上 | 替换为本课词汇 |
| 阅读站1 第6题 | monkey | 同上 | 替换为本课词汇 |
| 阅读站4 第3题 | 情境暗示duck（"会游泳"） | 间接引导，答案不在本课词汇 | 重写情境 |

### 检查方法

```bash
# 在项目根目录执行，查找所有题库文件中的 value/correct/distractor 字段
grep -n "\"value\"\|\"correct\"\|\"distractor\"" js/u1l1_coop.js
# 对照本课词汇表逐行核查
```

开发完成后必须做此检查，不能凭记忆判断。

## 二、题型设计规范

### 规则

**B 同学的任务必须考察语言知识，不能仅靠图片判断。**

具体要求：
- B 同学的操作界面显示英文单词，不显示动物图片（除非题型本身要求图文配对）
- B 的选项必须是需要语言处理才能判断的选项（如中文、英文单词），不能是直接视觉匹配

### 反例

早期设计中，听音接力的 B 步骤是"看图选中文"：

```
A 听音频 → 选动物图片
B 看A选的图 → 选对应中文
```

问题：B 的任务不需要任何英语知识，图→中文是视觉直觉，不涉及语言处理。这道题对 B 来说没有英语学习价值。

### 规范后的设计

```
A 听音频 → 选动物图片
B 看大号蓝色英文单词（如 "BEAR"） → 选对应中文
```

B 需要认识英文单词才能选出正确中文，这才是真正的词汇知识考察。

## 三、A/B 角色规范

### 规则1：必须按题目索引奇偶轮换

```javascript
// 正确写法：每道题计算当前谁是A
const isEven = window.currentQuestionIndex % 2 === 0;
window.currentCoopRoleA = isEven ? window.players[0] : window.players[1];
```

不能写死某一方永远是 A。

### 规则2：必须同步到 window.players，不能只用局部变量

```javascript
// 错误写法（bug根源）：
let players = ["小明", "小红"];  // 局部变量，其他函数拿不到

// 正确写法：
window.players = ["小明", "小红"];  // 全局可访问
```

U1L1 曾出现因为 `window.players` 未正确赋值，导致游戏内显示"A同学/B同学"而非真实姓名的 bug。

### 规则3：角色名字必须从 getPlayerShortName() 获取

```javascript
// 禁止写法：
const roleName = isRoleA ? "A同学" : "B同学";

// 正确写法：
const roleName = getPlayerShortName(isRoleA ? window.players[0] : window.players[1]);
```

所有显示角色名字的地方（提示文字、判断结果、支架提示）统一走 `getPlayerShortName()`，不硬编码。

## 四、支架系统规范

### 四级梯度（必须按顺序，不能跳级）

| 级别 | 支架类型 | 具体实现 | 禁止的替代方案 |
|------|---------|---------|-------------|
| 级1 | 同伴协商（零技术） | 黄色提示框"先和同学讨论一下" | 直接播放音频、直接给提示 |
| 级2 | Modified input | 分词慢速播放 + 词卡蓝色高亮 | 重复原速播放原音频 |
| 级3 | 语言线索 | 首字母提示（如 "b___"） | 减少选项数（construct shift） |
| 级4 | 多模态 | 图片 + 英文文字 + 分词播放同时呈现 | 直接显示答案 |

### 禁止事项

**禁止：直接减少选项（Construct Shift）**

减少选项等于改变了题目考查的内容——原本需要从5个中认出正确答案，变成从2个中猜。学生无需真实语言知识就能通过，造成虚假的"支架效果"。

**禁止：无修改重复播放音频（无信息增量）**

重复播放原速音频对学生没有帮助。第2级支架必须是 modified input：更慢的速度 + 视觉高亮同步 = 新的可理解性。

**必须：第1级支架不给技术线索，强制同伴交流**

如果第1次答错就立刻给技术线索，学生会等App提示而不是跟同学讨论。第1级只显示"先和同学讨论"的提示框，零技术介入。

## 五、常见 Bug 清单

U1L1 开发过程中遇到的 bug，新课程开发前必须逐一检查。

### Bug 1：window.players 未同步 → 名字显示为"A同学/B同学"

**现象**：游戏中角色提示显示"A同学"而非学生真实姓名。

**根因**：选人页赋值给局部变量 `let players`，但题目逻辑读取的是 `window.players`，两者不同步。

**修复**：选人页确认后，同时给 `window.players` 和局部变量赋值，并在题目逻辑中统一只读 `window.players`。

---

### Bug 2：sequence.length 判断提前跳题 → 干扰项点击无效

**现象**：听音排序题中，点击干扰项（错误选项）后题目立即跳走，无法选择。

**根因**：判断"排序完成"的逻辑是 `selectedSequence.length >= sequence.length`，而 `sequence` 长度包含了干扰项，导致点完第2个就误判完成。

**修复**：修改判断条件为"正确答案序列全部选满"，干扰项点错不计入序列长度。

---

### Bug 3：干扰项词汇超出本课范围 → 学生困惑

**现象**：学生在选项中看到不认识的词（monkey、duck），无法判断是对是错，产生混乱。

**根因**：题库设计时沿用上学期词汇作为干扰项，未对照本课词汇表检查。

**修复**：全面 grep 题库文件，将所有超出范围的词替换为本课5个词汇（bear/horse/bird/panda/rabbit）。

---

### Bug 4：模块卡片 onclick 被 JS 覆盖 → 点击无反应

**现象**：模块选择页的四张卡片（听力/阅读/写作/口语）点击无反应。

**根因**：`app.js` 中的旧初始化代码用 `addEventListener` 覆盖了 HTML 中写的 `onclick`，两个事件绑定冲突。

**修复**：删除 `app.js` 中的旧绑定代码，统一只在 HTML 里写 `onclick="showAdventureMap('listening')"` 等。

---

### Bug 5：listen_judge B 区域在 A 操作前剧透答案

**现象**：听音判断题中，B 的操作区域在 A 完成前就显示了正确答案的提示文字。

**根因**：`instruction` 参数在 B 等待状态下传入了包含答案线索的文字。

**修复**：B 等待时 `instruction` 传空字符串，A 完成后再填充 B 的操作说明。

---

### Bug 6：选错后卡死，无法重选

**现象**：选错答案后按钮变灰，但500ms后不恢复，学生卡死在错误状态。

**根因**：`createOptionGrid` 函数中按钮禁用逻辑没有配套的恢复逻辑。

**修复**：加500ms延迟后重新启用按钮，允许取消并重选。

---

### Bug 7：真实姓名未显示（同 Bug 1 的变体）

**现象**：部分题型（如合作建句）显示"A同学"，但听力题正常显示真实名字。

**根因**：合作建句的 `renderBuildSentence()` 函数有独立的角色名获取逻辑，没有走统一的 `getPlayerShortName()`。

**修复**：所有题型的角色名显示统一走 `getPlayerShortName(window.currentCoopRoleA)`，不各自实现。

## 2026-04-09 下午 新增经验

### Bug：分发表遗漏导致功能静默失效

**场景**：新增题型 `coop_build_sentence` 后，A完成步骤但B区永远灰色卡死，无报错。

**根因**：`renderCoopStepB` 内部用分发表（对象字面量）按 `question.type` 分发渲染函数，新题型忘记加入分发表 → 分发到 `undefined` → 静默失效。

**规则**：每新增题型，必须同时检查并更新所有分发表（`renderCoopStepA`、`renderCoopStepB`、`judgeStepA`、`judgeStepB`）。

---

### Bug：渲染函数未清除旧DOM导致内容翻倍

**场景**：`renderStepB_BuildSentence` 被多次调用（如重试），词块出现两份。

**根因**：函数只负责创建新DOM，未清除上一次渲染留下的 `.duo-word-grid`。

**规则**：所有 `renderStepX_*` 函数开头必须先 `querySelector` 找到已有容器并 `remove()`，再重新创建。

---

### Bug：异步竞争条件导致已完成题目被误重置

**场景**：听音排序，点完最后一个正确词后100ms内仍可点击词卡 → 误触干扰项 → `resetCards()` 重置，但 `setTimeout` 里的 `onStepBComplete` 照常执行 → 状态不一致。

**根因**：完成判断与UI锁定之间存在时间窗口。

**规则**：完成状态必须用布尔标志位（如 `sortCompleted`）在判断逻辑的第一行立即置 `true`，所有后续事件处理先检查标志位再执行。

---

### 设计决策：多邻国交互标准

- 答案区用下划线 slot，点词块自动填入，不用输入框
- 已填词块可点击取回（该词及之后同区词块全部返回词库）
- 不设撤销按钮，点击取回即撤销
- 正确：绿闪动画→自动进下一题；错误：红抖动画→重置让重选
