# App Data Architecture — Advisor Briefing

## Overview
Single table design: `app_behavior_log` captures all student interaction data per question.

## Data Fields (17 fields)

For each field, explain:
- What it records
- Which research objective it serves (Obj1-4)
- What analysis method uses it

Fields:
| Field | Description | Objective | Analysis Method |
|-------|-------------|-----------|-----------------|
| logId | Unique record identifier | — | Data integrity check |
| studentId | Student number (anonymized) | All | Individual trajectory analysis |
| partnerId | Partner's student number | Obj3 | Pair effect analysis |
| pairId | Pair identifier (e.g., "15-16") | Obj3 | Collaboration quality analysis |
| sessionDate | Timestamp when question was answered | Obj2/Obj3 | ITS time series, engagement frequency |
| lessonId | Lesson identifier (e.g., "U1L1") | Obj2/Obj4 | Cross-lesson comparison |
| module | Skill module: listening/reading/writing/speaking | Obj2 | Skill-specific development analysis |
| stationId | Station number within module | Obj2 | Difficulty progression analysis |
| questionType | Coop question type (e.g., coop_listen_relay) | Obj1 | Design effectiveness analysis |
| questionIndex | Question position in station (1-5) | — | Fatigue/position effect analysis |
| targetWord | Target vocabulary/sentence tested | Obj2 | **Item-level mastery tracking** |
| isCorrect | Whether answer was correct ("true"/"false") | Obj2/Obj4 | **Core accuracy metric** |
| asRoleA | Student's role in this question ("true"=A, "false"=B) | Obj3 | Role effect analysis |
| timeSpentMs | Response time in milliseconds | Obj3 | **Engagement depth, processing speed** |
| scaffoldLevelUsed | Scaffold level triggered (0=none, 1-4=levels) | Obj1/Obj3 | **Scaffolding effectiveness analysis** |
| retryCount | Number of retries before correct | Obj2 | **Learning curve analysis** |
| soeScore | Speech evaluation score (speaking module only) | Obj2 | **Oral proficiency development** |

## Analysis Methods Mapped to Objectives

### Obj1: App Design (Qualitative)
- **Method**: Design narrative + ADR documentation
- **Data used**: questionType, scaffoldLevelUsed
- **Analysis**: Describe how each design decision maps to theoretical framework (6 theories). Use scaffoldLevelUsed distribution to show scaffold system is functioning as designed.

### Obj2: English Development (Quantitative)
- **Method 1**: ITS (Interrupted Time Series) on weekly test scores
  - Data: Weekly paper test scores (separate Feishu table)
  - Analysis: Compare baseline period (8 weeks) vs intervention period (8 weeks)

- **Method 2**: Item-level mastery curves
  - Data: targetWord + isCorrect + sessionDate
  - Analysis: For each vocabulary item (bear, bird, panda, etc.), plot accuracy over time. Define mastery as 3 consecutive correct answers.

- **Method 3**: Oral proficiency trajectory
  - Data: soeScore + sessionDate
  - Analysis: Track speaking scores over time, compare early vs late performance

- **Method 4**: Module comparison
  - Data: module + isCorrect
  - Analysis: Compare accuracy across listening/reading/writing/speaking to identify skill-specific development patterns

### Obj3: Engagement & Self-efficacy (Mixed Methods)
- **Method 1**: Behavioral engagement indicators
  - Data: timeSpentMs + sessionDate
  - Analysis: Track average response time over weeks. Decreasing time + stable accuracy = growing fluency. Increasing time = deeper processing or disengagement (triangulate with observation).

- **Method 2**: Scaffolding effectiveness
  - Data: scaffoldLevelUsed + isCorrect + retryCount
  - Analysis: Calculate scaffold dependency ratio (% questions needing scaffold). Decreasing ratio over time = growing independence (ZPD expansion). Cross-reference with Bandura's mastery experience.

- **Method 3**: Persistence indicators
  - Data: retryCount + isCorrect
  - Analysis: Students who retry more AND eventually succeed show higher self-efficacy (mastery experience). Compare with satisfaction survey responses.

- **Method 4**: Role effect analysis
  - Data: asRoleA + isCorrect (same student across questions)
  - Analysis: Compare performance when student is A (English task) vs B (Chinese task). Role alternation ensures both skills are exercised.

- **Method 5**: Pair collaboration effect
  - Data: pairId + isCorrect + timeSpentMs
  - Analysis: Compare pair-level accuracy. Some pairs may perform better, indicating effective collaboration (Social Constructivism evidence).

### Obj4: Learning Outcomes (Quantitative)
- **Method**: ITS on weekly test scores + behavior log triangulation
  - Primary data: Weekly paper test scores (Feishu table)
  - Supporting data: isCorrect trends from behavior_log
  - Analysis: Show that in-app performance improvement correlates with test score improvement

## Triangulation Framework (5 data sources)

| Source | Type | What it proves |
|--------|------|----------------|
| App behavior_log | Process data (automatic) | Students used the app AND improved over time |
| Weekly paper tests | Outcome data (teacher-scored) | English proficiency actually improved |
| Classroom observations | Observational (teacher) | Classroom engagement increased |
| Student satisfaction survey | Self-report (student) | Students perceive the app as helpful |
| Teaching reflections | Reflective (teacher-researcher) | Teacher observed qualitative changes |

## Design Principles
1. **Data minimization**: Only 17 fields, each mapped to a specific research objective
2. **Single table**: Reduces technical complexity and failure points
3. **Offline-first**: localStorage queue ensures no data loss on weak network
4. **iOS 12 compatible**: No modern JS features, lightweight JSON (~500 bytes per record)
5. **Ethical compliance**: No personal information beyond anonymized student numbers
