# Merry English App 新方案：合作冒险设计

> **状态**: 📝 草稿（2026-04-09 开始）
> **定位**: 替代旧设计方案的新思路，旧方案保留不动
> **核心变化**: 从"两人轮流用的单人答题App"→"必须两人分工合作才能完成的冒险系统"

---

## 一、设计理念：为什么要重新设计？

### 旧方案的根本问题

1. **Social Constructivism 是贴标签**：两人共享设备只是物理层面，去掉一个人App照常运行。评审一句话打死："如果一个人也能完成所有任务，凭什么说这是建构主义的？"
2. **3P 几乎没有体现**：Play只有翻牌游戏和星星（表面游戏化），Problem只有少数scenario题，Project完全没有
3. **本质上跟市面单人答题App没有区别**

### 新方案的核心原则

> **每道题拆成2步：A做第1步，B做第2步。两人都看同一屏幕，但各自只能操作自己的部分。B的正确取决于A的选择。**

- ❌ 不需要藏信息（共享5.5寸屏做不到）
- ✅ 通过**角色分工**和**步骤依赖**创造合作必要性
- ✅ 蓝色区域=A操作，橙色区域=B操作（A完成后B才激活）

---

## 二、App整体结构

```
老师选课（单元/课时）
  ↓
模块选择页（听力 / 阅读 / 写作 / 口语）
  ↓
冒险地图（每个模块独立一张地图）
  ↓
站点1 → 站点2 → 站点3 → ... → 终点
  ↓
每个站点 = 一组合作任务（A步骤 → B步骤）
```

### 关键结构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 前测 | **取消** | 旧前测是单人模式，与合作设计割裂；用weekly quiz替代纵向追踪 |
| 模块 | **听说读写各自独立** | 每节课选一个模块，一周4课覆盖4模块 |
| 时间分配 | 老师10min → App 25min → 其他5min | 取消前测后，全部时间给合作任务 |
| 地图站点数 | 每模块3-5个站点 | 25min ÷ 5-8min/站点 ≈ 3-5站 |

## 三、六大理论的具体实现

### 理论1：Social Constructivism（社会建构主义）— ✅ 已解决

**实现方式**：蓝橙分工机制

每道题分为A步骤和B步骤：
- A做完后，B的区域才激活
- B的正确答案取决于A的选择
- 单独一个人无法完成任务

**设计检验标准**：把B拿掉，任务还能完成吗？如果不能→建构主义成立。

**具体形式（按题型）**：

| 合作形式 | 说明 | 适用题型 |
|---------|------|---------|
| **接力式** | A做第1步，结果传给B做第2步 | 听音接力、词义接力、接力填空 |
| **拼合式** | A有一半材料，B有另一半，合起来才完整 | 合作建句、合作拼词 |
| **轮流式** | AB交替操作，共同推进任务 | 听音排序、句子排序、翻牌配对 |
| **师徒式** | A先做示范，B模仿A | 口语跟读接力、看图说话 |

---

### 理论2：SLA 二语习得（i+1）

**实现方式**：自适应出题算法（保留旧方案的核心逻辑）

#### 设计层面（静态 i+1）
- 题目从词→句→语篇递进
- 中文翻译+图片+语音让输入可理解
- 题库设计时就嵌入了难度梯度

#### 运行时（动态 i+1）
- 冒险地图的站点难度递增：站点1=easy为主 → 站点3=hard为主
- 站点内的自适应：连续低支架答对3题→难度升级，连续高支架2题→难度降级
- 使用scaffoldLevel（不仅是对错）作为能力信号

#### 与旧方案的区别
- 旧方案：自适应在30题的线性流程中运作
- 新方案：自适应在冒险地图的站点间+站点内两层运作

---

### 理论3：CLT 认知负荷理论

**实现方式**：界面设计约束

| CLT策略 | App中的实现 |
|---------|-----------|
| 分段呈现 | 一屏一任务，不同时显示A和B的操作内容 |
| 难度递进 | 站点1→3难度递增，站点内easy→hard |
| 双编码 | 图片+文字+音频多模态呈现 |
| 降低外在负荷 | 中文翻译、蓝橙色块清晰区分AB角色 |
| 减少分心 | B区域在A完成前灰显，避免信息过载 |

---

### 理论4：ZPD + Scaffolding（最近发展区+支架）

**实现方式**：合作模式下的双重支架

#### 支架来源1：App系统支架（保留旧方案的三级支架逻辑）
- Level 0：一次答对（无支架）
- Level 1：隐性提示（如重播音频、显示图片）
- Level 2：半显性提示（缩到2选项、显示中文）
- Level 3：显性提示（最大线索+2选1，但永远不给答案）
- Level 4：跳题，回收到队列末尾

#### 支架来源2：同伴支架（Peer MKO）— 新增！
**这是新方案独有的理论亮点。**

在合作模式下，A和B自然形成同伴支架关系：
- **接力式任务**：A的选择结果帮B缩小范围（A选对了图，B看到图就知道选什么中文）
- **师徒式任务**：A跟读示范，B模仿A的发音（A=同伴MKO）
- **拼合式任务**：强学生自然会告诉弱学生"这个放这里"

**关键论点**：旧方案的支架只有"App→学生"一个方向。新方案增加了"学生→学生"方向——这正是Vygotsky说的MKO（More Knowledgeable Other）。

#### 合作模式下支架的特殊处理
- 支架在**B步骤**触发（A步骤不触发，因为A是先行者）
- A步骤错误 → 不立即判错，而是让B有机会纠正（两人一起重做）
- 这符合建构主义"错误是学习机会"的理念

---

### 理论5：Self-efficacy（自我效能感）

**实现方式**：多层次成功体验

| Bandura的四来源 | App中的实现 |
|----------------|-----------|
| **掌握体验** | 星星积分、站点通关标记✅、进度条 |
| **替代体验** | 看到搭档答对（"A答对了！"），增强"我也行"的信念 |
| **言语说服** | 鼓励消息（"连对3题！""你们配合得真好！"） |
| **生理状态** | 冒险地图的视觉进度（已完成站点变绿），减少焦虑 |

#### 新方案独有的Self-efficacy增强
- **合作成功比个人成功更有激励性**：两人一起通关的成就感 > 一个人做对一道题
- **弱学生也有贡献感**：拼合式任务中，弱学生放的那一块也是成功的一部分
- **失败是共同的**：答错时系统说"你们没配合好"而不是"你错了"，减少个人挫败感

### 理论6：3P（Project / Problem / Play）— 三层实现

**这是新方案最重要的创新，也是旧方案最大的缺陷修复。**

#### 3P 三层架构

```
┌────────────────────────────────────────┐
│  单元层 (Unit)  → Project-based       │
│  完成4课后解锁单元项目，产出可见成果    │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │  站点层 (Station) → Problem-based│  │
│  │  每模块最后1站 = 情境问题任务     │  │
│  │                                  │  │
│  │  ┌────────────────────────────┐  │  │
│  │  │  题目层 (Task) → Play-based│  │  │
│  │  │  游戏化交互=学习过程本身    │  │  │
│  │  └────────────────────────────┘  │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘
```

#### Play-based（题目层）

**原则**：游戏机制本身就是学习过程，不是学习的包装。

| Play题型 | 怎么玩 | 为什么是Play而非Exercise |
|---------|--------|------------------------|
| 翻牌配对 | AB轮流翻牌，找图-词配对 | 翻牌过程=强化词义配对记忆 |
| 听音排序 | AB轮流放，按听到的顺序排 | 排序过程=强化听力序列记忆 |
| 拼图闯关 | 每答对一题，拼图露出一块 | 好奇心驱动，答题获得视觉奖励 |
| 合作拼词 | AB各放一半字母拼出单词 | 拼字母过程=强化拼写记忆 |
| 抢答竞速 | 限时内AB合作答最多题 | 时间压力+合作=自动化+团队感 |

**检验标准**：去掉游戏机制，学习效果会变吗？如果会变差→真Play-based。

#### Problem-based（站点层）

**原则**：每个模块的最后一个站点是情境问题任务——有条件限制+有信息差+需要合作判断。

**与普通题目的区别**：
- 普通题：选正确答案（只有对错）
- Problem题：根据条件做决策（有多种可能，需要判断）

**具体设计示例**：

| 模块 | Problem任务 | A的角色 | B的角色 |
|------|-----------|---------|---------|
| 听力 | "你朋友不吃肉，帮他点餐" | A听菜单音频 | B看条件"不吃肉"，一起排除选项 |
| 阅读 | "帮迷路的小动物找到家" | A读动物描述 | B看地图上的位置，一起匹配 |
| 写作 | "给新同学写欢迎卡" | A选问候语 | B选自我介绍句，合作组成卡片 |
| 口语 | "在动物园当导游介绍动物" | A说动物名称 | B说完整介绍句 |

**条件限制是关键**：不是"选正确答案"，而是"在限制条件下做最佳选择"。

#### Project-based（单元层）

**原则**：完成整个Unit（4课）后，解锁一个需要综合运用所学内容的小项目，产出可见成果。

**Unit项目设计**：

| Unit | 主题 | 项目任务 | 产出成果 |
|------|------|---------|---------|
| Unit 1 | Animals | 动物园地图：把动物拖到正确位置+给每个动物写一句话 | 一张完整的动物园地图 |
| Unit 2 | My Family | 家庭树：A选家庭成员图片，B写介绍句，合作完成家庭树 | 一张家庭海报 |
| Unit 3 | Food | 餐厅菜单：AB合作设计一份菜单（选食物图+写英文名+标价格） | 一份菜单 |
| Unit 4 | Clothes | 穿搭秀：给角色搭配衣服+用英语描述穿着 | 一个穿搭方案 |

**项目的合作模式**：
- A负责视觉部分（选图、拖放、排版）
- B负责语言部分（选词、写句、配文字）
- 两人必须合作，各出一半
- 最终成果保存在App中，老师和其他同学可以看到

**项目的理论价值**：
- **Social Constructivism**：两人共同建构一个产出
- **SLA**：综合运用听说读写
- **Self-efficacy**：产出可见的成果→"我们做到了！"
- **3P-Project**：唯一真正的Project-based体现

## 五、支架系统（全题型四级支架）

### 5.1 四级支架通用原则

> 核心约束：**任何级别都不能减少选项数量**（保护 construct validity，不改变任务的认知构念）

| 级别 | 触发时机 | 内容形式 | 颜色 |
|-----|---------|---------|------|
| Level 1 | 第1次答错 | 黄色横幅「🤝 先和同学讨论一下再选！」，零技术线索 | 黄色 #FFF3CD |
| Level 2 | 第2次答错 | 题型专属的 Modified Input（见下表） | 蓝色 #E3F2FD |
| Level 3 | 第3次答错（未超过强制跳过阈值） | 语言知识线索（首字母/中文翻译/图片） | 蓝色 |
| Level 4 | 强制跳过时设定 | 多模态全开（图+文+音） | 蓝色 |

**支架累加显示**：新级别出现时旧级别保留，学生能看到完整的帮助历史。

### 5.2 各题型 Level 2-4 具体实现

| 题型 | Level 2 | Level 3 | Level 4 |
|------|---------|---------|---------|
| coop_listen_relay | 慢速重播音频（0.8x速度） | 显示图片下方中文标注 | 图+音+中文三通道 |
| coop_listen_judge | 慢速重播整句音频 | 显示句子文字 | 文字+音频+中文翻译 |
| coop_word_relay | 图片轻微放大+边框高亮 | 显示英文词首字母 | 图+首字母+中文 |
| coop_flip_match | 已翻过的牌半透明显示位置记忆提示 | 高亮一张正确配对牌的边框（闪烁） | 直接显示正确配对位置 |
| coop_build_sentence | 显示句子中文翻译 | 第一个词块位置高亮提示 | 中文翻译+词块顺序数字提示（1,2,3...） |
| coop_relay_fill | 显示图片（如有）+播放音频 | 显示空格对应的中文提示词 | 图+中文+音频 |
| coop_spell_word | 慢速播放单词音频 | 正确字母格高亮（闪烁） | 音频+正确字母高亮+中文 |
| coop_sentence_sort | 播放完整句子音频 | 高亮第一个词的词块 | 音频+中文翻译+第一词高亮 |
| coop_read_relay | 重播系统示范音频 | 显示音标辅助 | 音频+音标+文字放大 |
| coop_picture_speak | 显示中文提示 | 显示句型框架（This is a ___） | 中文+框架+参考词 |
| coop_dialogue | 显示情境中文翻译 | 显示对话开头词提示 | 中文+开头词+完整句型参考 |

### 5.3 coop_listen_sort（特殊案例，独立内置支架）

`coop_listen_sort` 已有独立四级支架（errorCount 计数器），不走通用支架框架：
- 错1次：黄色横幅「先和同学讨论一下，再试试」
- 错2次：蓝色+分词慢速播放+词卡蓝色高亮
- 错3次：橙色+首字母提示（如 "b___"）
- 错4次：绿色+词卡显示英文文字+分词播放

### 5.4 自适应掌握算法（Adaptive Mastery Loop）

#### 站内循环队列

**数据结构：**

```js
questionQueue: [],          // 当前待做题目（题目索引数组）
questionQueue_original: [], // 原始题目列表（用于星星计算）
retryCountMap: {},          // 每题重试次数 {index: count}
forceSkipped: [],           // 强制跳过的题目索引
scaffoldLevels: {},         // 每题当前支架级别 {index: level}
```

**答题逻辑：**
- 站点开始时，按难度配比排入队列（easy→medium→hard）
- **答对**：从队头移除
- **答错（retryCount < 3）**：retryCountMap[index]++，scaffoldLevels[index] 升1级（最高4），题目移到队尾
- **答错（retryCount >= 3）**：标记为 forceSkipped，scaffoldLevel=4，从队列移除
- **站点完成**：questionQueue.length === 0

#### 站间难度调整

| 上站星星数 | 下站 easy 比例 | medium | hard |
|-----------|--------------|--------|------|
| 3星 | 20% | 50% | 30% |
| 2星 | 40% | 40% | 20% |
| 1星 | 60% | 30% | 10% |

#### 星星计算标准

| 条件 | 结果 |
|-----|------|
| 0 题被强制跳过 + 平均 scaffoldLevel < 1 | ⭐⭐⭐ 三星 |
| 0 题被强制跳过（用了支架） | ⭐⭐ 二星 |
| 有题目被强制跳过 | ⭐ 一星 |

### 5.5 "合作重做"机制（同伴协商层）

当AB合作答错时：
1. 第1-2次错 → 显示「一起商量重来吧！」提示框（可包含题目 hint）
2. 支架升级后题目回到队尾等待
3. 第3次仍错 → 强制跳过，标记 forceSkipped

**理论价值**：第一次错误是"合作学习机会"（建构主义），支架升级是 ZPD 动态响应，强制跳过防止挫败感（Self-efficacy 保护）。

| 学生满意度 | 每单元 | 量化+质性 | 参与度评估 |

### App自动收集的数据字段

**站点级数据：**

| 字段 | 说明 |
|------|------|
| 搭档组合 | A是谁+B是谁 |
| 模块 | listening/reading/writing/speaking |
| 站点编号 | 第几个站点 |
| 各站点得分 | 星星数（1/2/3） |
| A步骤正确率 | A的贡献 |
| B步骤正确率 | B的贡献 |
| 合作重做次数 | 多少次触发了"一起重来" |

**每题级数据（questionDetailLog 数组）：**

```js
{
    questionType: string,     // 题型（如 "coop_listen_relay"）
    difficulty: string,       // 难度 easy / medium / hard
    retryCount: number,       // 本题重试次数
    finalScaffoldLevel: number, // 最终使用的支架级别（0-4）
    wasForceSkipped: boolean, // 是否被强制跳过
    wasCorrect: boolean,      // 最终是否答对
    timeSpent: number,        // 用时（毫秒）
    questionIndex: number     // 题目在站点中的索引
}
```

**数据价值**：retry次数+支架级别是学习行为的**过程性指标**，反映了真实的努力程度和知识掌握过程，比单纯的对错率更有研究价值（符合 SLA 研究对 process data 的重视）。

#### 新功能：四级支架系统（听音排序题）

**理论依据**：Wood et al. contingent scaffolding + Vygotsky 社会建构 + Long 互动假说

| 错误次数 | 支架类型 | 具体实现 | 理论依据 |
|---------|---------|---------|---------|
| 错1次 | 强制同伴协商 | 黄色提示框"先和同学讨论"，无技术线索 | Vygotsky：社会建构优先于技术介入 |
| 错2次 | Modified input | 分词慢速播放，每词对应词卡蓝色高亮 | Long 互动假说：可理解输入要修正，不要重复 |
| 错3次 | 语言线索 | 显示首字母如"b___"，不改变选项数 | 保持 construct validity，语言知识线索 |
| 错4次 | 多模态输入 | 词卡图片下方显示英文文字 + 分词播放 | 图文音三通道，降低认知负荷 |

#### 学术讨论记录

**i+1 和 ZPD 在题型设计中的具体体现**：
- 听音接力 = i（可理解输入，图片辅助）
- 听音判断 = i+1（略超现有水平，需判断对错）
- 情境听力 = i+2（情境+条件判断，最高认知要求）

**批判原支架设计的5个问题**：
1. 任务支架 ≠ 语言支架（缩减选项是降低任务难度，不是帮助语言理解）
2. 重复播放 ≠ modified input（重复相同输入不产生新的可理解性）
3. 减少选项 = construct shift（改变了题目考查的认知构念）
4. 战略性失败风险（学生可能故意答错以获得更多提示）
5. 技术介入打断同伴协作（App给出线索时，两人停止讨论）

**新支架设计原则**：
- 同伴先于技术（第一次错误先触发同伴协商，不直接给技术线索）
- Modified input 而非重复（慢速分词 + 高亮 = 真正的输入修正）
- 语言知识线索而非任务操作线索（首字母提示帮助词汇识别，不改变题目构型）
