# 理论驱动设计指南

> **用途**：开发出题、设计App功能时的理论参考和检查清单
> **来源**：2026-04-09 学术讨论成果
> **关联文档**：`新方案_App合作冒险设计.md`

---

## 一、六理论逻辑框架

```
【WHY层】为什么这样学？
  └─ Social Constructivism → 学习在社会互动中建构

【HOW层】怎么让合作学习有效？
  ├─ ZPD + Scaffolding → 双支架（App支架 + 同伴支架）
  ├─ i+1 → 自适应控制输入难度
  └─ CLT → 合作分步 = 分担认知负荷

【WHAT层】学习体验长什么样？
  ├─ 3P → Play/Problem/Project 三层任务
  └─ Self-efficacy → 合作成功积累效能感

【贯穿层】AI = 技术引擎
  └─ 让上面三层在App里真正运行
     - SOE → 实时口语评测
     - 自适应算法 → 实现 i+1
     - 智能支架系统 → 实现 ZPD
     - 合作机制 → 实现 Social Constructivism
```

### 核心论点

- AI 不是独立理论，是让六个理论从纸面变成可运行系统的技术手段
- 六个理论不是并列标签，是 WHY→HOW→WHAT 的因果链
- 从学生角度：他们体验到的是一个"智能的、会自动调节的合作学习环境"

## 二、每个理论的设计检查清单

### 1. Social Constructivism（WHY层）

**核心要求**：每道题必须两人合作才能完成，去掉一个人任务就无法完成。

检查项：
- [ ] 这道题拆成了A步骤和B步骤吗？
- [ ] B的操作依赖A的结果吗？
- [ ] 把B拿掉，A能独立完成整个任务吗？（如果能→设计失败）

四种合作形式（出题时选一种）：

| 形式 | 说明 | 适用题型 |
|------|------|---------|
| 接力式 | A做第1步，结果传给B做第2步 | 听音接力、词义接力、接力填空 |
| 拼合式 | A有一半材料，B有另一半 | 合作建句、合作拼词 |
| 轮流式 | AB交替操作 | 听音排序、句子排序、翻牌配对 |
| 师徒式 | A示范，B模仿 | 口语跟读接力、看图说话 |

### 2. i+1（HOW层）— 管"选什么题"

**核心要求**：给学生的题目难度刚好超出当前水平一点点。

检查项：
- [ ] 题库里的题目有难度标签吗？（easy/medium/hard）
- [ ] 自适应算法能根据答题表现选下一题吗？
- [ ] 站点间难度递增了吗？（站点1=easy为主 → 站点3=hard为主）
- [ ] 有中文翻译/图片/音频让输入"可理解"吗？

**i+1 工作时机**：题目之前（从题库选题）和一组题做完后（调整下一组难度）

### 3. ZPD + Scaffolding（HOW层）— 管"给多少帮助"

**核心要求**：学生做题过程中，根据表现实时调整支架等级。

#### App支架（系统→学生）

| 等级 | 触发条件 | 支架内容 |
|------|---------|---------|
| Level 0 | 一次答对 | 无支架 |
| Level 1 | 犹豫/首次错 | 隐性提示（重播音频、显示图片） |
| Level 2 | 再次错 | 半显性提示（缩到2选项、显示中文） |
| Level 3 | 连续错 | 最大提示（最大线索+2选1，但不给答案） |
| Level 4 | 仍然错 | 跳题，回收到队列末尾 |

#### 同伴支架（学生→学生）— 新方案独有

- 接力式：A的选择结果帮B缩小范围
- 师徒式：A示范发音，B模仿
- 拼合式：强学生自然指导弱学生

检查项：
- [ ] 支架在B步骤触发（A步骤不触发，A是先行者）
- [ ] A步骤错误时不立即判错，给B机会纠正
- [ ] 这道题的合作形式能产生同伴支架吗？

**ZPD 工作时机**：题目之中（答题过程中实时调整）

### 4. CLT 认知负荷理论（HOW层）

**核心要求**：合作分步本身就是分担认知负荷；界面设计降低外在负荷。

检查项：
- [ ] AB分步是否真的分担了认知负荷？（A处理一半信息，B处理另一半）
- [ ] 一屏只有一个任务吗？（不同时显示A和B的操作内容）
- [ ] B区域在A完成前灰显了吗？
- [ ] 用了多模态呈现吗？（图片+文字+音频）
- [ ] 中文翻译在需要时显示了吗？
- [ ] 蓝橙色块清晰区分AB角色了吗？

### 5. 3P — Play/Problem/Project（WHAT层）

#### Play（题目层）

**核心要求**：游戏机制本身就是学习过程，不是包装。

检查项：
- [ ] 去掉游戏机制，学习效果会变差吗？（如果不会→不是真Play）
- [ ] 游戏过程是否就是在练习目标技能？

#### Problem（站点层）

**核心要求**：最后一站是情境问题，有条件限制，需要判断。

检查项：
- [ ] 有真实情境吗？（如餐厅点餐、问路）
- [ ] 有条件限制吗？（如"为素食朋友点餐"）
- [ ] 答案不是唯一的，需要判断？
- [ ] 与普通题的区别：不是选对错，而是做决策

#### Project（单元层）

**核心要求**：完成一个单元后，两人合作产出可见成果。

**落地形式：拼海报**

学生英语极弱、不会打字，所以产出形式是**从预设素材中选择和组合**（拖拽操作）。

示例：Unit学了水果词汇 → Project是"开一个水果店"
1. App提供素材库：水果图片、价格标签、英文单词卡、店名模板
2. A负责选水果图片和摆放位置
3. B负责拖对应的英文标签和价格
4. 两人合作拼出一张"水果店海报"
5. 完成后可截图保存/展示

**为什么拖拽素材算"真实产出"**：对小学三年级、英语极弱的学生来说，从素材中做选择和组合就是他们当前水平能做到的最高层次产出。Project难度也要符合i+1，不能超出能力范围。

检查项：
- [ ] 提供了足够的预设素材吗？（图片、单词卡、标签）
- [ ] A和B的分工明确吗？（一人选素材，一人配文字）
- [ ] 产出物跟本单元学的内容相关吗？
- [ ] 完成后有展示/保存机制吗？

### 6. Self-efficacy 自我效能感（WHAT层）

**核心要求**：通过多层次成功体验建立"我能行"的信念。

| Bandura四来源 | App实现 |
|--------------|--------|
| 掌握体验 | 星星积分、站点通关标记、进度条 |
| 替代体验 | 看到搭档答对（"A答对了！"） |
| 言语说服 | 鼓励消息（"连对3题！""你们配合得真好！"） |
| 生理状态 | 冒险地图视觉进度（已完成站点变绿） |

检查项：
- [ ] 合作成功的反馈是"你们"而不是"你"？
- [ ] 失败反馈是"你们没配合好"而不是"你错了"？
- [ ] 弱学生在任务中也有明确的贡献感？

## 三、配对设计

**方式**：完全随机配对

**理由**：
- 方法论最干净，避免研究者干预偏差
- 如果随机配对下仍然有效，证明App设计足够robust

**弱+弱配对的兜底机制**：
- 认知层面：App三级支架系统兜底（ZPD）
- 情感层面：合作本身降低焦虑、互相带动参与（Self-efficacy）
- 论点：弱+弱时，Social Constructivism的价值从"认知建构"转为"情感支持"

## 四、关键论证（应对答辩追问）

### Q: 为什么不是六个理论贴标签？
A: 六个理论是WHY→HOW→WHAT的因果链。Social Constructivism决定了"必须合作"，ZPD+i+1+CLT解决了"合作怎么有效"，3P+Self-efficacy定义了"体验和动力"。去掉任何一层，系统都不完整。

### Q: CLT不就是基本UI设计吗？
A: 不只是UI。合作分步机制本身就是CLT——AB各处理一半认知负荷。这是用合作来分担认知负荷，是本设计独有的CLT实现。

### Q: 弱+弱配对时Peer MKO不成立？
A: 承认弱+弱时认知建构主要靠App支架。但合作仍在情感层面起作用：降低个人挫败感、互相带动参与。这对应Self-efficacy理论。App设计要robust到不依赖完美配对。

### Q: AI-assisted体现在哪？
A: AI贯穿整个系统：开发层（AI设计课件、分析数据、写代码）+ 运行层（SOE实时评测、自适应算法、智能支架）。从学生角度，他们体验到的是一个会自动调节的智能学习环境。

### Q: 合作比独立学习更好的证据？
A: 强+弱/中+中配对：Peer MKO产生认知建构价值。弱+弱配对：合作产生情感支持价值。文献支持+课堂观察中的真实案例。
