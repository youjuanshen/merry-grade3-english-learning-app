# 合作冒险App 设计理念说明
（Collaborative Adventure App: Design Rationale）

> **用途**：向导师解释App设计思路的学术说明文档
> **版本**：2026-04-09（基于U1L1 MVP实现）
> **学生对象**：泰国小学三年级，英语水平极弱，每节课25分钟App时间，两人共用一台设备

---

## 0. 导师核心问题答案（答辩关键）

> **导师问题**（2026-04-09）："How do you apply all selected theories to design app, and conduct implementation plan? Show me how you blend all selected theories. Concrete ideas to help all committees see the same picture."

### 核心答案：一道题，六个理论同时工作

**以听音接力（听力模块站1）为例：**
A 听到 "bear" 的音频，从4张动物图片中选出熊 → B 看到A选的图，从4个中文词中选出"熊"

| 此刻发生了什么 | 对应理论 | 如何体现 |
|-------------|---------|---------|
| 音频+图片同时呈现，A能理解这个词 | **SLA i+1** | 多模态让输入从 i+2 降到 comprehensible i+1 |
| A选完图，B的区域才激活；去掉B任务无法完成 | **Social Constructivism** | 结构性合作依赖，非物理共用设备 |
| A只处理音-图，B只处理图-义，各负担一半认知负荷 | **CLT 认知负荷** | 合作分步本身就是认知负荷分担机制 |
| A答错不立即判错，B可纠正；第一次错强制同伴协商 | **ZPD + Scaffolding** | 同伴支架优先于技术支架，动态介入 |
| 两人一起答对 → "你们太棒了！" | **Self-efficacy** | 合作成功 = 掌握体验（Bandura来源一） |
| 翻牌/排序游戏机制本身就是学习过程 | **3P Play-based** | 去掉游戏机制，学习效果变差→真Play |

**这就是 blend：不是六个理论贴六个标签，而是同一个交互瞬间，六个理论同时在工作，缺任何一个设计就会崩塌。**

### 答辩英文说法

> "Let me show you one question — and how all six theories work simultaneously in that single interaction. [展示上表] This is what I mean by blending: every design decision is driven by multiple theories at once, not tagged after the fact. If I remove the cooperative mechanism, Social Constructivism collapses. If I remove the scaffolding, ZPD has no operational form. If I remove the game mechanic, 3P Play-based disappears. The theories are not parallel labels — they are a system."

## 1. 理论框架 (Theoretical Framework)

本App的设计以六个理论为基础，形成 **WHY → HOW → WHAT** 的因果链，而非并列贴标签。

```
【WHY层】为什么这样学？
  └─ Social Constructivism（社会建构主义）→ 学习在社会互动中建构

【HOW层】怎么让合作学习有效？
  ├─ Vygotsky ZPD + Scaffolding（最近发展区+支架）→ 双支架
  ├─ Krashen i+1（可理解输入假说）→ 自适应难度控制
  └─ CLT, Cognitive Load Theory（认知负荷理论）→ 合作分步分担负荷

【WHAT层】学习体验长什么样？
  ├─ 3P Framework（Play / Problem / Project）→ 三层任务架构
  └─ Bandura Self-efficacy（自我效能感）→ 合作成功积累信念
```

### 1.1 Vygotsky ZPD + 同伴协作 (Peer Collaboration as MKO)

**理论要点**（中文）：Vygotsky 的最近发展区（Zone of Proximal Development）指出，学习者在独立能力之上存在一个"借助他人帮助可以达到"的区间。关键词：MKO（More Knowledgeable Other，"更有知识的他者"）可以是教师，也可以是同伴。

**英文术语**：ZPD, Peer MKO, contingent scaffolding (Wood et al., 1976)

**本App的实现**：
旧设计中，唯一的支架方向是 App → 学生（单向技术支架）。新设计中，通过 A/B 角色分工，强学生自然承担 MKO 角色——例如接力式任务中，A 的选择结果作为信息输入传递给 B，帮助 B 缩小判断范围。这是真正的同伴支架，而非师生支架的替代品。

**设计验证标准**：把 B 同学拿掉，A 能独立完成整个任务吗？如果能 → 合作机制设计失败。

---

### 1.2 SLA 二语习得理论（完整循环）

**理论来源**：Krashen（输入假说）、Long（交互假说）、Swain（输出假说）

**理论要点**（中文）：SLA 有效学习不是单一输入，而是一个完整循环：
**Input（可理解输入）→ Interaction/Negotiation（互动协商）→ Feedback（及时反馈）→ Pushed Output（推动产出）**

本App四个模块正好对应这条 SLA 完整链条。

**英文术语**：comprehensible input, i+1, affective filter (Krashen); negotiation of meaning, modified input (Long); pushed output, output hypothesis (Swain)

#### SLA 完整链条在 App 中的体现

| SLA 环节 | 理论来源 | App 中的实现 |
|---------|---------|------------|
| **Input 可理解输入** | Krashen i+1 | 听力模块：音频 + 图片 + 中文多模态呈现；题目难度从 i 到 i+1 递进 |
| **Interaction 互动协商** | Long 交互假说 | A/B 合作机制：两人必须协商才能完成任务；错误触发同伴协商（第一级支架） |
| **Feedback 及时反馈** | Long + ZPD | 答对/答错即时动画；支架提示是 modified input（修正输入，非重复输入） |
| **Pushed Output 推动产出** | Swain 输出假说 | 写作模块（书面产出）+ 口语模块（口头产出）：强迫学生从理解转向主动产出 |

#### Swain 输出假说的具体体现

Swain (1985) 认为仅有输入不足以习得语言，学习者需要被"推动"产出（pushed output），才能：
1. **注意到语言缺口**（noticing the gap）：试图说出/写出时发现自己不知道怎么表达
2. **检验语言假设**（hypothesis testing）：尝试表达 → 得到反馈 → 修正

本App中的 Pushed Output：
- **写作模块**：合作建句、接力填空、合作拼词——学生不是选答案，而是主动产出语言形式
- **口语模块**：看图说话、情境对话——从模仿（跟读）到自主表达（说完整句），逐步推动输出

#### i+1 静态与动态实现

| 层级 | 具体体现 |
|------|---------|
| 静态 i+1（题库设计层） | 题目从词 → 句 → 语篇递进；站点1以 easy 为主，站点3以 hard 为主 |
| 动态 i+1（运行时自适应） | 自适应队列：上一站⭐⭐⭐ → 本站难题比例提升；有题被跳过 → 本站降低难度配比 |
| 可理解性保障 | 中文翻译 + 图片 + 音频多模态呈现，确保输入 comprehensible |

U1L1 听力模块 i+1 链：
- 听音接力（站1）= i（图片辅助，纯识别）
- 听音判断（站2）= i+1（意义理解 + 判断）
- 听音排序（站3）= i+1（序列记忆 + 合作）
- 情境听力（站4）= i+2（推理 + 决策）

### 1.3 Wood et al. Contingent Scaffolding（情境性支架）

**理论要点**（中文）：Wood, Bruner & Ross (1976) 提出的 contingent scaffolding（情境性支架）强调支架应根据学习者的实时表现动态调整：表现好则撤架（fading），表现差则增援——关键是支架不能改变任务本身的认知构念（construct validity）。

**英文术语**：contingent scaffolding, fading, construct validity, modified input (Long, 1983)

**本App的四级支架系统（以听音排序题为例）**：

| 错误次数 | 支架类型 | 具体实现 | 理论依据 |
|---------|---------|---------|---------|
| 错1次 | 强制同伴协商 | 黄色提示框"先和同学讨论"，零技术线索 | Vygotsky：社会建构优先于技术介入 |
| 错2次 | Modified input | 分词慢速播放 + 每词对应词卡蓝色高亮 | Long 互动假说：修正输入而非重复 |
| 错3次 | 语言知识线索 | 显示首字母如 "b___"，不改变选项数 | 保持 construct validity，考察词汇识别而非猜测 |
| 错4次 | 多模态输入 | 词卡图片下方显示英文文字 + 分词播放 | 图文音三通道，降低认知负荷 |

**与"直接减少选项"的本质区别**：减少选项是 construct shift（改变了任务考查的认知构念），学生不需要语言知识就能凭猜测通过。四级支架设计中任何一级都不减少选项数，始终考察真实的语言知识。

---

### 1.4 CLT, Cognitive Load Theory（认知负荷理论）

**理论要点**（中文）：Sweller (1988) 的认知负荷理论指出，工作记忆容量有限，学习设计应控制外在认知负荷（extraneous load），优化内在认知负荷（intrinsic load），促进生成性认知负荷（germane load）。本App中 CLT 的实现不只是界面设计，更是通过合作分步从结构上分担认知负荷。

**英文术语**：extraneous load, intrinsic load, germane load, split-attention effect, dual coding

**本App的实现**：
- **合作分步本身就是 CLT 的核心体现**：A 处理一半认知负荷（如选图片），B 处理另一半（如选中文义），两人合起来才完成一个完整的认知任务。这是用合作机制来分担认知负荷，而非仅靠界面设计
- **降低外在负荷**：中文翻译 + 图片 + 音频多模态呈现，蓝橙色块清晰区分 AB 角色，B 区在 A 完成前灰显（避免信息过载）
- **双编码（Dual Coding）**：图片 + 文字 + 音频三通道同时呈现，符合 Paivio (1991) 双编码理论，降低单一通道负荷
- **分段呈现**：一屏一任务，A 步骤和 B 步骤不同时显示，避免 split-attention effect

## 3. A/B 合作机制设计

### 3.1 为什么要角色轮换？

**设计决策**：每道题按题目索引奇偶自动轮换 A/B 角色（通过 `window.currentCoopRoleA` 控制）。

**理论依据**：
- 防止固定角色导致"强学生永远做A（先手）"，造成弱学生长期处于被动 B 位，剥夺其语言输出机会
- 轮换保证两人都有平等机会担任 initiator（发起者）和 responder（回应者）
- 符合 CLT 中双向交际（two-way communication）的要求

**已实现**：U1L1 中所有题型按奇偶索引自动切换，角色名从 `getPlayerShortName()` 获取真实学生姓名。

### 3.2 为什么 B 看英文单词而非图片？

**设计决策**：在听音接力题中，B 同学的操作界面显示大号蓝色英文单词，不显示动物图片。

**理论依据（Construct Validity）**：
- 若 B 看图选中文，B 不需要任何英语知识就能完成任务（图-中文直接对应，绕开目标语言）
- 这破坏了题目的 construct validity——我们声称在测量英语词汇知识，但实际测量的是图片识别能力
- B 看英文单词选中文，强制 B 进行词汇层面的语言处理（英文词→中文义），这才是真正的语言知识考察

### 3.3 同伴支架 vs 技术支架的优先级

**设计原则**：同伴支架优先，技术支架兜底。

| 优先级 | 支架类型 | 触发时机 |
|--------|---------|---------|
| 1（最高） | 同伴协商（Peer negotiation） | 第1次错误：强制提示"先和同学讨论" |
| 2 | Modified input | 第2次错误：慢速分词+高亮 |
| 3 | 语言线索 | 第3次错误：首字母提示 |
| 4 | 多模态输入 | 第4次错误：图文音三通道 |

**理论依据**：技术支架过早介入会打断同伴交流（"App给线索了，我不用和同学商量了"），损害 Social Constructivism 的实现。第1级支架零技术线索，强制社会协商发生。

## 4. 四级支架系统（以听音排序为例）

### 4.1 设计原则

1. **同伴优先**：第1级零技术线索，强制同伴先商量
2. **Modified input 而非重复播放**：重复播放同一音频不产生新的可理解性（Long, 1983）；慢速分词+高亮才是真正的输入修正
3. **语言知识线索而非任务操作线索**：首字母 "b___" 帮助词汇识别，不改变任务结构
4. **不改变任务构型（no construct shift）**：四级支架始终保持相同的选项数，不因为难以完成就减少选项

### 4.2 四级对应的理论依据

| 级别 | 支架内容 | 理论依据 | 理论来源 |
|------|---------|---------|---------|
| 级1 | 强制同伴协商，无技术介入 | 社会建构优先于技术介入 | Vygotsky (1978) |
| 级2 | Modified input：分词慢播+词卡高亮 | 互动假说：修正输入提升可理解性 | Long (1983) |
| 级3 | 首字母线索 "b___" | 语言知识支架，保持 construct validity | Wood et al. (1976) |
| 级4 | 图文音多模态 | 双编码理论，降低认知负荷 | Paivio (1991); CLT |

### 4.3 与"直接给答案"的本质区别

**Fading（撤架）vs Gaming（攻略）**：

- "直接减少选项"本质上是改变任务，学生通过猜测而非语言知识通过——这是 gaming（钻制度空洞），不是 learning
- 真正的支架（fading 原则）是：帮助学生用语言知识解题，而不是降低解题所需的语言知识门槛
- 四级支架中，即使到第4级（多模态），学生仍需要自己选择正确词卡——语言处理过程保持完整

### 4.4 自适应掌握循环（Adaptive Mastery Loop）

#### 机制说明

本App实现了一套**站内循环队列 + 站间难度调整**的自适应掌握算法：

**站内循环：**
- 错题不直接丢弃，而是升级支架后**放回队列末尾**，等待再次练习
- 每题最多重试3次；第3次仍错则标记为"强制跳过"，不再入队
- 站点完成判定：所有题答对或被强制跳过（动态队列清空）

**支架联动：**
- 每次答错：`scaffoldLevel[题目索引]++`（上限4）
- 重新遇到该题时，自动恢复并显示已累积的支架
- 保证"每次遇到错题，帮助都比上次更充分"

**强制成功体验：**
- 即使一道题答错3次，最终仍以"Level 4全支架"状态完成（而非标记为失败）
- 站点结算界面：「[n]道题需要复习，下次加强！」——问题被延后处理，而非立即判定失败

#### 理论依据

| 机制 | 理论支撑 |
|------|---------|
| 错题回队列反复练习 | **ZPD（最近发展区）**：在ZPD内反复练习直到穿越；支架随错误次数递增，模拟Vygotsky的contingent scaffolding |
| 强制成功体验（至少能用Level 4完成） | **Self-efficacy（自我效能感）**：Bandura指出掌握体验是效能感最核心来源；即使困难，完成任务本身也积累效能 |
| 支架升级让难题变为可解 | **i+1（Krashen）**：支架把原本的i+2（超出范围）拉回到i+1（可理解）范围内 |
| 退出条件：队列清空而非题目数量 | **Mastery Learning**：每道题都要达到"答对或用尽支架"，不机械完成固定数量 |

#### 站间调整（Dynamic Difficulty Adjustment）

完成一站后，系统根据本站星星数调整下一站的题目难度配比：

| 本站表现 | 下一站配比 | 理论意图 |
|---------|-----------|---------|
| 3星（无跳题+低支架） | easy 20% / medium 50% / hard 30% | 学生已在舒适区，推向i+1 |
| 2星（无跳题，有支架） | easy 40% / medium 40% / hard 20% | 维持当前挑战水平 |
| 1星（有跳题） | easy 60% / medium 30% / hard 10% | 降低挑战，回到可理解输入范围 |

#### 数据价值

**过程性数据（Process Data）**是本研究的核心数据优势：

- `retryCount`：反映学生真实遇到困难的次数（比"正确率"更敏感）
- `finalScaffoldLevel`：反映完成任务需要的支持程度（量化ZPD宽度）
- `wasForceSkipped`：识别完全超出学生能力范围的词汇/句型
- `timeSpent`：每题耗时反映流畅度（SLA的自动化程度指标）

这四个字段组合，可以在单题粒度追踪每个学生的**学习行为轨迹**，而非只有对错的结果数据。这是本App相对于普通练习App的研究价值所在。

## 5. i+1 跨模块完整对应表（Four-Module i+1 Chain）

> 以下说明每个模块内部的 i+1 纵向链。每一站相对于上一站都是 i+1——链式递进，而非单一跳跃。

### 5.1 为什么站1是 i（而非 i-1）？

站1的多模态设计保证了 comprehensible input：
- 听力站1：听音频同时看4张图片 → 图片是 i 的来源，即使完全不认识单词也能完成
- 阅读站1：看图选英文词 → 图片锚定意义，英文词是视觉辨认任务
- 写作站1：所有词块预先提供 → 不需要记忆词汇，只需理解语序
- 口语站1：听系统音频跟读 → 有完整模型音频，只需模仿

**理论依据**：多模态支撑把输入变成 comprehensible（i），合作要求（必须和搭档配合）把任务变成 i+1。

### 5.2 四模块 i+1 纵向链

| 模块 | 站1（i） | 站2（i+1） | 站3（i+2相对站2） | 站4（最高层） |
|------|---------|-----------|-----------------|-------------|
| **听力** | 听音认图（音-图配对，视觉辅助） | 听句判断（意义理解，判断真伪） | 听词排序（序列记忆，工作记忆要求） | 听情境决策（推理+条件判断） |
| **阅读** | 看图选词（视觉锚点，图-词配对） | 翻牌配对（记忆+词义匹配，无持续视觉提示） | 句子排序（句法理解，语序重建） | 情境阅读（信息提取+条件推理） |
| **写作** | 合作建句（词块已给，理解语序即可） | 接力填空（语义选择，需理解句意填词） | 合作拼词（字母粒度，需知拼写序列） | 情境写作（语用组合，有条件限制） |
| **口语** | 跟读接力（有模型音频，模仿即可） | 看图说话（无音频，自主输出） | 情境对话（角色扮演，交际运用） | — |

### 5.3 横向 i+1（同一题型内的难度梯度）

每个题型内部有 easy / medium / hard 三档。以听音接力为例：

| 难度 | 设计差异 | i+1 体现 |
|------|---------|---------|
| easy | 4个选项均为本课核心词汇，音频清晰 | 基础词汇识别 |
| medium | 干扰项增加，或图片相似度更高 | 精细辨别要求 |
| hard | 句子级音频（如 "This is a bear."），而非单词 | 句子级理解 |

自适应算法根据支架使用量动态调整下一题难度，实现运行时 i+1。

## 6. Self-Efficacy 自我效能感（逐题型触发点）

> Bandura (1997) 的自我效能感理论指出，效能感来自四个来源：掌握体验、替代体验、言语说服、生理状态。以下列出本App每个来源的**具体触发时机和内容**。

### 6.1 掌握体验（Mastery Experience）— 最核心来源

| 触发时机 | 具体反馈 | 设计意图 |
|---------|---------|---------|
| 一次答对（Level 0，无需支架） | ⭐ 星星动画 + "你们太棒了！" | 强化"我们有能力做到"的信念 |
| 连续3题无支架答对 | 特殊动画 + "你们配合得真好！连对3题！" | 积累掌握体验，强化合作效能感 |
| 站点通关 | 站点变绿✅ + 得星数显示 | 可见的阶段性成功 |
| 完成整个模块 | 冒险地图完成动画 | 单元级成就感 |

**关键设计**：成功反馈语言使用"你们"（集体），而非"你"（个人）——强化合作效能感，而非个人效能感。

### 6.2 替代体验（Vicarious Experience）

| 触发时机 | 具体反馈 |
|---------|---------|
| A步骤答对，B步骤激活时 | "【A姓名】答对了！轮到你了 →" |
| 搭档在支架提示后答对 | "你们一起解决了！" |

**理论价值**：看到搭档成功 → "他能做到，我也能" → 替代体验强化自身效能感。

### 6.3 言语说服（Verbal Persuasion）

| 场景 | 反馈文案 |
|-----|---------|
| 用了支架但最终答对 | "有点难，但你们坚持住了！" |
| 合作重做后答对 | "商量一下就解决了，这才是团队！" |
| 答错时 | "你们没配合好，再来一次吧！"（非"你错了"） |

**关键**：失败归因于"合作方式"（可改变），而非"能力不足"（固定），保护效能感不受挫败侵蚀。

### 6.4 生理状态（Physiological State）— 降低焦虑

| 设计元素 | 降焦虑机制 |
|---------|----------|
| 冒险地图视觉进度（已完成站点变绿） | 可见进度 → 减少"不知道还有多少"的焦虑 |
| B区在A完成前灰显 | 明确"现在不是我的回合" → 减少等待焦虑 |
| 支架不惩罚（不扣星、不倒退） | 试错无代价 → 降低对失败的生理恐惧 |
| 合作失败归因于"配合"而非"能力" | 保护心理安全感 |

## 7. 3P 框架（Play / Problem / Project）

> **理论定位**：3P 是三种学习方式（Play-based / Problem-based / Project-based），三者在本App中同时存在、相互交织。三者的交汇点是**意义协商与共同建构（Meaning Negotiation & Co-construction）**——这是学习真正发生的核心机理。

### 为什么 3P 交汇处能产生意义协商？

- **Play** 提供心理安全环境：游戏规则让学生有理由尝试、允许犯错，降低焦虑（affective filter↓）
- **Problem** 提供真实目的：有条件限制的情境任务让学生有理由协商——"这道题没有唯一对答案，我们得商量"
- **Project** 提供可见产出：共同产出物让学生有动力协商——"这是我们一起做的东西"

三者缺一，协商的动机就会消失：只有 Play 没有 Problem → 游戏变成机械练习；只有 Problem 没有 Play → 焦虑过载；只有 Project 没有 Play/Problem → 产出没有过程。

### 3P 在 App 中的操作化映射

本App将 3P 三种学习方式映射到三个层级，让每节课同时包含三种 P：

| 3P 类型 | 操作化层级 | 覆盖范围 |
|---------|----------|---------|
| Play-based | 题目层（每道题） | 所有游戏化题型 |
| Problem-based | 站点层（每模块最后一站） | 四个情境任务站 |
| Project-based | 单元层（完成4课后） | 单元合作项目（MVP待实现）|

### 7.1 Play-based（题目层）

**核心检验标准**：去掉游戏机制，学习效果会变差吗？如果会 → 真Play-based。

| 题型 | 游戏机制 | 为什么是真Play |
|------|---------|--------------|
| 听音排序 | AB轮流放词卡，排列顺序 | 轮流放置过程 = 强化序列记忆，不能改为"选一个答案" |
| 翻牌配对 | AB轮流翻牌找配对 | 翻牌记忆过程 = 词义强化，偷看就破坏了学习机制 |
| 合作拼词 | AB各放一半字母拼出单词 | 拼字母过程 = 拼写记忆，不能改为"直接选整词" |
| 合作建句 | 词块拖入下划线槽，两色拼合 | 拖放拼合过程 = 句法感知，视觉化语序 |

### 7.2 Problem-based（站点层）

每个模块**最后一站**是情境问题任务：有条件限制 + 需要合作判断 + 无唯一正确答案路径。

| 模块 | 情境问题 | 条件限制 | 与普通题的区别 |
|------|---------|---------|--------------|
| 听力站4 | 帮朋友点喜欢的食物 | "朋友喜欢大的动物" | 不是选对错，是在限制条件下做决策 |
| 阅读站4 | 帮迷路动物找到家 | 动物有特定描述特征 | 需阅读条件+匹配地图，信息整合 |
| 写作站4 | 给新同学写欢迎卡 | 特定问候场景 | 选词+组合，有语用目的 |
| 口语站3 | 在动物园当导游 | 角色扮演场景 | 真实交际目的驱动语言输出 |

### 7.3 Project-based（单元层）— MVP阶段待实现

完成整个Unit（4课）后，解锁单元项目。产出形式为**素材拼合**（适合英语极弱的学生，拖拽操作而非自由输入）：

| Unit | 主题 | 项目任务 | A的分工 | B的分工 | 产出 |
|------|------|---------|---------|---------|------|
| Unit 1 | Animals | 动物园地图 | 选动物图片+摆位置 | 拖配英文名标签 | 完整动物园地图 |
| Unit 2 | Family | 家庭树 | 选家庭成员图片 | 拖对应英文称谓 | 家庭海报 |
| Unit 3 | Food | 餐厅菜单 | 选食物图片+摆版面 | 拖英文名+价格标签 | 菜单一份 |
| Unit 4 | Clothes | 穿搭秀 | 给角色选衣服 | 拖穿搭描述词卡 | 穿搭方案 |

**为什么素材拼合也算"真实产出"**：对三年级英语极弱学生，在素材中做选择和组合是其当前水平能达到的最高层次语言产出（符合 i+1，不超出能力范围）。Project 难度本身也要符合 i+1，不能因为是"项目"就大幅提高要求。

## 8. 站点数量设计依据

### 设计原则

> 各模块站点数由该课知识点密度决定，而非固定统一。研究者根据课本内容手动配置，在 `stationConfig` 字段中指定。

**答辩说法**：站点数量是内容适应性设计（content-adaptive design）的一部分——词汇量少的模块减少站点以匹配认知负荷（CLT），保证25分钟内完成学习目标。

### U1L1 配置依据

| 模块 | 站点数 | 配置理由 |
|------|-------|---------|
| 听力 | 4站 | 5个核心词+6个句型，听力维度需要充分练习 |
| 阅读 | 4站 | 词义认读+句法理解需要分步覆盖 |
| 写作 | 4站 | 建句→填空→拼词→情境，写作技能递进明显 |
| 口语 | 3站 | 口语产出难度高，学生水平极弱，3站已覆盖模仿→表达→交际三层 |

## 9. 词汇选择原则

### 5.1 严格限定在本课5个词汇

U1L1 核心词汇：**bear / horse / bird / panda / rabbit**

**规则**：题库所有选项（包括干扰项）必须在本课词汇列表内，不出现生词。

**理论依据**：
- Krashen i+1 要求输入"略超"当前水平，而非"大幅超出"——生词干扰项属于 i+3 以上，超出可理解范围
- 生词干扰项让学生因为"不认识"而排除，而不是因为"理解了目标词"而选择——测量的是生词识别能力而非目标词掌握程度，破坏 construct validity

### 5.2 干扰项也必须是已学词汇

**已实现**：2026-04-09 完成词汇清理，将代码中所有 monkey / duck 替换为本课5个词汇（monkey / duck 是上学期知识点，不属于 U1L1）。

**检查方法**：对所有题目 value 字段 grep 检查，逐一对照词汇表核查。
